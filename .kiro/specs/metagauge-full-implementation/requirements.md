# Requirements Document

## Introduction

MetaGauge is a Web3 analytics platform that provides actionable growth intelligence for blockchain protocol founders, product managers, and investors. The current application has basic authentication, contract onboarding, on-chain indexing, and a basic dashboard. This document defines the full implementation requirements to deliver the complete vision: wallet intelligence, competitive analytics, real-time alerts, AI-driven growth advice, export/sharing, SDK access, and category benchmarking — all built on a PostgreSQL backend with role-based access control.

## Glossary

- **System**: The MetaGauge platform (backend + frontend)
- **Indexer**: The on-chain data ingestion service that continuously fetches transactions, events, and function calls from registered smart contracts
- **Wallet**: An externally-owned blockchain account (EOA) that interacts with a registered smart contract
- **Contract**: A smart contract registered by a user for analytics tracking
- **Cohort**: A group of wallets that first interacted with a contract within the same time window (e.g., same week)
- **D1/D7/D30/D90**: Day-1, Day-7, Day-30, Day-90 retention — the fraction of a cohort that returned on that day
- **LTV**: User Lifetime Value — the total fees/value generated by a wallet over its lifetime
- **RPAW**: Revenue Per Active Wallet — total revenue divided by the count of active wallets in a period
- **Churn**: A wallet that has not interacted with the contract beyond the configured inactivity threshold
- **Whale**: A wallet ranked in the top 10 by total transaction volume for a given contract
- **Bot**: A wallet exhibiting automated, non-human interaction patterns
- **Alert**: A system-generated notification triggered when a metric crosses a configured threshold
- **Competitor**: A smart contract or protocol added by a user to their competitive watchlist
- **Opportunity Score**: A 0–100 score representing the potential value of a strategic action derived from competitive analysis
- **Share Token**: A time-limited, cryptographically random token that grants read-only access to a dashboard
- **AI Growth Advisor**: The proactive AI agent that generates scheduled briefings and reactive advice using Google Gemini
- **Briefing**: A scheduled AI-generated summary (daily, weekly, or monthly) delivered in-app and via email
- **Benchmark**: A statistical summary (average, percentile) of a metric across all contracts in the same category
- **Category**: A classification for a smart contract — one of: DeFi, NFT, DAO, Gaming, Infrastructure
- **Admin**: A user role with full read/write access including user management
- **Analyst**: A user role with read/write access to analytics and alerts but no user management
- **Viewer**: A user role with read-only access to dashboards and reports
- **Progress**: The percentage completion (0–100) of an ongoing indexing or analysis job
- **blockRange**: The inclusive range of block numbers (start, end) covered by an indexing job

## Requirements

---

### Requirement 1: Real-Time Indexing Progress Accuracy

**User Story:** As a user who has just onboarded a contract, I want the progress bar to reflect the actual backend processing state, so that I can trust the UI and know when indexing is truly complete.

#### Acceptance Criteria

1. WHEN the Indexer processes each batch of transactions, THE Indexer SHALL emit a progress update within 2 seconds of completing that batch.
2. THE System SHALL report indexing progress as a percentage value between 0 and 100, where the reported value is within 5 percentage points of the actual backend processing completion percentage at all times.
3. WHEN the Indexer completes a transaction batch, THE System SHALL update the `indexingProgress` field in the user record and the `progress` field in the analysis record atomically before emitting the WebSocket event.
4. WHEN the frontend receives a progress WebSocket event, THE Dashboard SHALL render the updated progress value within 500 milliseconds.
5. IF the WebSocket connection is unavailable, THEN THE System SHALL fall back to polling the `/api/onboarding/status` endpoint at 5-second intervals to retrieve the current progress value.

---

### Requirement 2: Block Range Persistence

**User Story:** As a user viewing my dashboard, I want to see the block range that was indexed, so that I understand the data coverage of my analytics.

#### Acceptance Criteria

1. WHEN the Indexer completes an indexing job, THE Indexer SHALL write the `blockRange` object (containing `start` and `end` block numbers) to both `results.summary.blockRange` and `metadata.blockRange` in the analysis record.
2. WHEN the Dashboard fetches the default contract data, THE System SHALL return the `blockRange` from `metadata.blockRange` in the API response.
3. IF no completed analysis exists for a contract, THEN THE System SHALL return `null` for `blockRange` and THE Dashboard SHALL display "Not yet indexed" in place of block range values.

---

### Requirement 3: PostgreSQL Database Migration

**User Story:** As a platform operator, I want the system to use PostgreSQL for persistent storage, so that data is durable, queryable, and scalable beyond file-based limits.

#### Acceptance Criteria

1. THE System SHALL store all users, contracts, analyses, alerts, cohort data, competitor data, AI advice, share tokens, and benchmarks in a PostgreSQL database.
2. WHEN the server starts, THE System SHALL verify the PostgreSQL connection and all required tables exist before accepting requests.
3. IF the PostgreSQL connection fails on startup, THEN THE System SHALL log the error and exit with a non-zero status code.
4. THE System SHALL execute all database queries using parameterized statements to prevent SQL injection.
5. WHEN migrating from file-based storage, THE Migration_Script SHALL preserve all existing user, contract, and analysis records without data loss.

---

### Requirement 4: Role-Based Access Control

**User Story:** As an Admin, I want to assign roles to team members, so that I can control who can modify settings versus who can only view data.

#### Acceptance Criteria

1. THE System SHALL support exactly three roles: Admin, Analyst, and Viewer.
2. WHEN a new user registers, THE System SHALL assign the Admin role by default.
3. WHEN an Admin invites a team member, THE System SHALL allow the Admin to assign any of the three roles to the invited user.
4. WHILE a user holds the Viewer role, THE System SHALL reject any write requests (POST, PUT, DELETE) to analytics configuration endpoints with HTTP 403.
5. WHILE a user holds the Analyst role, THE System SHALL reject any requests to user management endpoints with HTTP 403.
6. WHEN a request arrives at a protected endpoint, THE Auth_Middleware SHALL verify the user's role before executing the handler.
7. THE Dashboard SHALL show or hide UI controls based on the authenticated user's role, such that Viewers see no edit or delete buttons.

### Requirement 5: Wallet Classification Engine

**User Story:** As a protocol founder, I want wallets automatically classified into segments (New, Active, Churned, Whale, Bot, High-risk), so that I can understand my user base without manual analysis.

#### Acceptance Criteria

1. THE WalletClassificationEngine SHALL classify every wallet that has interacted with a registered contract into exactly one primary segment: New, Active, Churned, Whale, Bot, or High-risk.
2. WHEN classifying wallets, THE WalletClassificationEngine SHALL apply segment rules in the following priority order: Bot → High-risk → Whale → New → Active → Churned.
3. THE WalletClassificationEngine SHALL classify a wallet as New if its first interaction with the contract occurred within the last 7 days.
4. THE WalletClassificationEngine SHALL classify a wallet as Active if it has interacted with the contract within the last 30 days and does not qualify as New, Whale, Bot, or High-risk.
5. THE WalletClassificationEngine SHALL classify a wallet as Churned if its most recent interaction with the contract was more than 30 days ago.
6. THE WalletClassificationEngine SHALL classify a wallet as Whale if it ranks in the top 10 wallets by total transaction volume for that contract.
7. THE WalletClassificationEngine SHALL classify a wallet as Bot if it meets any of the following heuristics: more than 100 transactions in any 24-hour window, or the same function called with identical input data more than 5 times within 60 seconds.
8. THE WalletClassificationEngine SHALL classify a wallet as High-risk if its failed transaction rate exceeds 40% of its total transactions.
9. WHEN the same wallet is re-classified on a subsequent run, THE WalletClassificationEngine SHALL produce the same segment for the same input data (deterministic output).
10. THE WalletClassificationEngine SHALL run the full classification pass for a contract on a schedule no less frequent than once every 24 hours.
11. THE System SHALL expose the wallet segment counts and per-wallet segment assignments via the `/api/analysis/:contractId/wallet-segments` endpoint.

### Requirement 6: Wallet Metrics Calculation

**User Story:** As a growth lead, I want to see LTV, RPAW, and revenue concentration metrics, so that I can understand the economic health of my protocol.

#### Acceptance Criteria

1. THE MetricsCalculator SHALL calculate User Lifetime Value (LTV) for each wallet as the sum of all gas fees paid plus the total ETH/token value transferred to the contract by that wallet.
2. THE MetricsCalculator SHALL calculate Revenue Per Active Wallet (RPAW) as total contract revenue in the period divided by the count of wallets classified as Active in that period.
3. THE MetricsCalculator SHALL calculate Revenue Concentration as the percentage of total contract revenue attributable to the top 10 wallets by volume.
4. WHEN a previously Churned wallet makes a new interaction with the contract, THE WalletClassificationEngine SHALL flag that wallet as Reactivated and record the reactivation timestamp.
5. THE System SHALL expose LTV, RPAW, Revenue Concentration, and reactivation events via the `/api/analysis/:contractId/wallet-metrics` endpoint.

---

### Requirement 7: Cohort Retention Analysis

**User Story:** As a product manager, I want to see D1, D7, D30, and D90 cohort retention rates, so that I can measure how well the protocol retains users over time.

#### Acceptance Criteria

1. THE CohortCalculator SHALL group wallets into weekly cohorts based on the calendar week of their first interaction with the contract.
2. FOR ALL cohorts, THE CohortCalculator SHALL calculate D1 retention as the count of wallets in the cohort that made at least one interaction on day 1 after their first interaction, divided by the cohort size.
3. FOR ALL cohorts, THE CohortCalculator SHALL calculate D7 retention as the count of wallets in the cohort that made at least one interaction between day 6 and day 8 after their first interaction, divided by the cohort size.
4. FOR ALL cohorts, THE CohortCalculator SHALL calculate D30 retention as the count of wallets in the cohort that made at least one interaction between day 28 and day 32 after their first interaction, divided by the cohort size.
5. FOR ALL cohorts, THE CohortCalculator SHALL calculate D90 retention as the count of wallets in the cohort that made at least one interaction between day 87 and day 93 after their first interaction, divided by the cohort size.
6. THE CohortCalculator SHALL store cohort retention results in the `cohort_retention` table with columns: contract_id, cohort_week, cohort_size, d1_retained, d7_retained, d30_retained, d90_retained, calculated_at.
7. WHEN cohort retention is recalculated for the same contract and cohort_week, THE CohortCalculator SHALL update the existing record rather than insert a duplicate.
8. THE System SHALL expose cohort retention data via the `/api/analysis/:contractId/cohort-retention` endpoint, returning all cohorts sorted by cohort_week descending.

### Requirement 8: Bot Detection

**User Story:** As a protocol founder, I want the system to automatically detect bot wallets, so that my retention and engagement metrics reflect genuine human users.

#### Acceptance Criteria

1. THE BotDetectionService SHALL evaluate every wallet against the bot heuristics defined in Requirement 5 (AC 7) after each indexing cycle.
2. THE BotDetectionService SHALL produce a false positive rate below 5% when evaluated against a labeled dataset of known human wallets.
3. WHEN a wallet is flagged as Bot, THE System SHALL exclude that wallet from all retention, LTV, and RPAW calculations by default.
4. WHERE a user has enabled manual bot review, THE System SHALL present flagged wallets for user confirmation before permanently classifying them as Bot.
5. THE System SHALL expose the list of bot-flagged wallets and their triggering heuristics via the `/api/analysis/:contractId/bot-wallets` endpoint.

### Requirement 9: Alerts Engine

**User Story:** As a founder, I want to receive automatic alerts when key metrics cross thresholds, so that I can respond to problems before they become critical.

#### Acceptance Criteria

1. THE AlertEngine SHALL evaluate all configured alert conditions for all contracts on a schedule no less frequent than once per hour.
2. THE AlertEngine SHALL trigger a Retention Drop alert WHEN the most recent D7 retention value for a contract falls below the user-configured threshold (default: 15%).
3. THE AlertEngine SHALL trigger a Whale Exit alert WHEN any wallet classified as Whale has not interacted with the contract for 7 or more consecutive days.
4. THE AlertEngine SHALL trigger a Revenue Spike alert WHEN the daily revenue for a contract increases by more than the user-configured threshold percentage compared to the previous day (default: 30%).
5. THE AlertEngine SHALL trigger a Revenue Dip alert WHEN the daily revenue for a contract decreases by more than the user-configured threshold percentage compared to the previous day (default: 30%).
6. THE AlertEngine SHALL trigger a Bot Surge alert WHEN the percentage of total interactions attributable to Bot-classified wallets exceeds the user-configured threshold (default: 20%).
7. THE AlertEngine SHALL trigger a Churn Spike alert WHEN the weekly churn rate increases by more than the user-configured threshold percentage compared to the previous week (default: 20%).
8. IF an alert condition is evaluated and the threshold is not crossed, THEN THE AlertEngine SHALL NOT create an alert record for that evaluation.
9. WHEN an alert is triggered, THE AlertEngine SHALL deliver the alert to the affected user via WebSocket within 10 seconds of detection.
10. THE System SHALL store all triggered alerts in the `alerts` table with fields: id, user_id, contract_id, alert_type, severity, message, metric_value, threshold, triggered_at, acknowledged_at, is_read.
11. WHEN a user acknowledges an alert, THE System SHALL record the acknowledged_at timestamp and set is_read to true.
12. THE System SHALL expose alert CRUD operations via `/api/alerts` (GET list), `/api/alerts/:id/acknowledge` (PATCH), and alert threshold configuration via `/api/alerts/config` (GET/PUT).

---

### Requirement 10: Alert Threshold Configuration

**User Story:** As an Analyst, I want to configure custom thresholds for each alert type, so that alerts are relevant to my protocol's specific context.

#### Acceptance Criteria

1. THE System SHALL allow users with Admin or Analyst roles to configure a numeric threshold for each of the five alert types: Retention Drop, Whale Exit inactivity days, Revenue Spike/Dip percentage, Bot Surge percentage, and Churn Spike percentage.
2. WHEN a user saves an alert threshold configuration, THE System SHALL validate that the threshold value is a positive number within the allowed range for that alert type.
3. IF a user has not configured a custom threshold, THEN THE AlertEngine SHALL use the default threshold values specified in Requirement 9.
4. THE System SHALL persist alert threshold configurations per user per contract in the database.

### Requirement 11: Competitor Management

**User Story:** As a founder, I want to add and manage competitor contracts in my watchlist, so that I can track how my protocol compares to the competition.

#### Acceptance Criteria

1. THE System SHALL allow a user to add a competitor by providing a smart contract address and chain, with an optional display name.
2. WHEN a competitor is added, THE System SHALL validate the contract address format for the specified chain before saving.
3. WHEN a competitor is added, THE Indexer SHALL begin background indexing of the competitor contract within 60 seconds.
4. THE System SHALL allow a user to remove a competitor from their watchlist, which SHALL stop future indexing of that competitor contract.
5. THE System SHALL allow a user to organize competitors into named groups.
6. THE CompetitorSuggestionEngine SHALL auto-suggest up to 5 competitor contracts based on the user's contract category and wallet overlap with other indexed contracts.
7. THE System SHALL expose competitor CRUD operations via `/api/contracts/:id/competitors` (GET, POST, DELETE).

### Requirement 12: Competitive Intelligence Dashboard

**User Story:** As a product manager, I want a side-by-side comparison of my protocol versus competitors, so that I can identify where I'm winning and where I'm falling behind.

#### Acceptance Criteria

1. THE CompetitiveDashboard SHALL display a side-by-side comparison table showing the following metrics for the user's contract and each tracked competitor: active wallets, D7 retention, daily revenue, 30-day growth rate, and churn rate.
2. THE CompetitiveDashboard SHALL display a leaderboard ranking all tracked competitors by each metric, with the user's contract position highlighted.
3. THE CompetitiveDashboard SHALL display 7-day, 30-day, and 90-day historical trend lines for all competitors on a single chart per metric.
4. THE CompetitiveDashboard SHALL display a scatter plot (landscape positioning map) with acquisition rate on the X-axis and retention rate on the Y-axis, with one data point per tracked contract.
5. WHEN competitor indexing data is unavailable, THE CompetitiveDashboard SHALL display "Indexing..." for that competitor's metrics rather than an error state.

### Requirement 13: Competitive Alerts

**User Story:** As a founder, I want to be alerted when a competitor makes a significant move, so that I can respond strategically in time.

#### Acceptance Criteria

1. THE AlertEngine SHALL trigger a Competitor Retention Surge alert WHEN a tracked competitor's D7 retention increases by more than 15 percentage points within any 7-day window.
2. THE AlertEngine SHALL trigger a Competitor User Acquisition Spike alert WHEN a tracked competitor's new wallet count increases by more than 50% within any 24-hour window.
3. THE AlertEngine SHALL trigger a Whale Migration alert WHEN a wallet classified as Whale on a competitor contract makes its first interaction with the user's contract.
4. THE AlertEngine SHALL trigger a TVL Overtake alert WHEN a tracked competitor's total transaction volume surpasses the user's contract total transaction volume for the first time.
5. THE AlertEngine SHALL trigger a Momentum Shift alert WHEN a tracked competitor's 7-day growth rate exceeds the user's contract 7-day growth rate by a factor of 2 or more for 7 consecutive days.
6. IF a competitive alert condition is not genuinely crossed (i.e., the threshold comparison is false), THEN THE AlertEngine SHALL NOT fire the alert.

### Requirement 14: Opportunity Scoring

**User Story:** As a growth lead, I want the system to surface ranked opportunities based on competitive analysis, so that I know where to focus my team's effort.

#### Acceptance Criteria

1. THE OpportunityScorer SHALL calculate a Feature Gap score (0–100) for each function signature used by at least one competitor but absent from the user's contract, based on the adoption rate of that function across all tracked competitors.
2. THE OpportunityScorer SHALL calculate a User Overlap score (0–100) for wallets that are active on competitor contracts but have never interacted with the user's contract, weighted by those wallets' LTV on the competitor.
3. THE OpportunityScorer SHALL calculate a Retention Play score (0–100) for competitors that have high D1 retention but low D30 retention, representing the opportunity to capture their churned users.
4. THE OpportunityScorer SHALL recalculate all opportunity scores after each competitor indexing cycle.
5. THE System SHALL expose the ranked opportunity list via `/api/analysis/:contractId/opportunities`, sorted by score descending.

### Requirement 15: CSV Export

**User Story:** As an analyst, I want to export transactions, wallet segments, and cohort data as CSV files, so that I can perform custom analysis in spreadsheet tools.

#### Acceptance Criteria

1. THE System SHALL provide a CSV export endpoint for transactions at `GET /api/analysis/:contractId/export/csv?type=transactions`.
2. THE System SHALL provide a CSV export endpoint for wallet segments at `GET /api/analysis/:contractId/export/csv?type=wallet-segments`.
3. THE System SHALL provide a CSV export endpoint for cohort retention data at `GET /api/analysis/:contractId/export/csv?type=cohort-retention`.
4. WHEN a CSV export is requested, THE System SHALL return a file whose rows contain exactly the same data values as those displayed in the corresponding dashboard table for the same contract and time range.
5. WHEN a CSV export is requested, THE System SHALL set the `Content-Type` header to `text/csv` and the `Content-Disposition` header to `attachment; filename="[type]-[contractAddress]-[date].csv"`.
6. IF the requested data set is empty, THEN THE System SHALL return a CSV file containing only the header row.

### Requirement 16: PDF Report Generation

**User Story:** As a founder, I want to generate an investor-ready PDF report, so that I can share protocol performance with investors without manual formatting.

#### Acceptance Criteria

1. THE ReportGenerator SHALL produce a PDF report containing: contract name, key metrics summary (active wallets, D7 retention, RPAW, revenue concentration), a 30-day growth chart, cohort retention table, and competitive position summary.
2. WHEN a PDF report is requested, THE System SHALL generate and return the PDF within 30 seconds.
3. THE System SHALL expose PDF generation at `GET /api/analysis/:contractId/export/pdf`.
4. THE PDF report SHALL contain only data values that match the values currently displayed on the dashboard for the same contract.

### Requirement 17: Shareable Read-Only Dashboard Links

**User Story:** As a founder, I want to generate a shareable link to my dashboard, so that investors or advisors can view my metrics without needing an account.

#### Acceptance Criteria

1. WHEN a user requests a share link, THE System SHALL generate a cryptographically random token of at least 32 bytes and store it in the `share_tokens` table with an expiry timestamp.
2. THE System SHALL set the default share token expiry to 7 days from creation.
3. WHEN a share token is accessed, THE System SHALL verify the token exists and has not expired before returning dashboard data.
4. IF a share token has expired, THEN THE System SHALL return HTTP 410 Gone.
5. THE System SHALL expose share token creation at `POST /api/analysis/:contractId/share` and read-only dashboard access at `GET /share/:token`.
6. WHEN a share link is accessed, THE SharedDashboard SHALL display the same metrics as the owner's dashboard but SHALL NOT display any edit controls, alert configurations, or competitor management options.
7. THE System SHALL allow a user to revoke a share token at any time, after which the token SHALL immediately return HTTP 410 Gone.

### Requirement 18: AI Growth Advisor — Proactive Advice

**User Story:** As a founder, I want the AI to proactively surface growth advice based on my metrics, so that I receive actionable insights without having to ask.

#### Acceptance Criteria

1. THE AIGrowthAdvisor SHALL generate proactive advice when any of the following conditions are detected: D7 retention below category average, competitor acquisition spike, feature gap detected, churn rate increasing week-over-week, RPAW declining, or bot activity exceeding 20%.
2. WHEN THE AIGrowthAdvisor generates advice, THE advice message SHALL reference the specific metric name and its current value that triggered the advice.
3. THE AIGrowthAdvisor SHALL store each generated advice item in the `ai_advice` table with fields: id, user_id, advice_type, content, trigger_metric, trigger_value, delivered_at, feedback, implemented, created_at.
4. THE AIGrowthAdvisor SHALL deliver proactive advice to the user via the in-app notification system within 60 seconds of generation.
5. THE System SHALL expose the advice list via `GET /api/advice` and individual advice items via `GET /api/advice/:id`.

### Requirement 19: AI Scheduled Briefings

**User Story:** As a founder, I want to receive daily, weekly, and monthly AI briefings, so that I stay informed about my protocol's performance without logging in every day.

#### Acceptance Criteria

1. THE BriefingScheduler SHALL generate a Daily Brief for each user at the user's configured delivery time (default: 08:00 UTC), containing: key metric changes from the previous day, one actionable insight, and any new competitor movements.
2. THE BriefingScheduler SHALL generate a Weekly Strategy briefing every Monday, containing: a deep dive on one strategic area, competitive landscape changes, and a recommended focus for the coming week.
3. THE BriefingScheduler SHALL generate a Monthly Board Summary on the first day of each month, containing: investor-ready metrics summary, progress against goals, risks, and opportunities.
4. WHEN a briefing is generated, THE EmailService SHALL deliver it to the user's registered email address within 5 minutes of generation.
5. WHEN a briefing is generated, THE System SHALL also make it available in-app via `GET /api/briefings`.
6. WHERE a user has disabled email delivery, THE BriefingScheduler SHALL still generate briefings and make them available in-app only.

### Requirement 20: AI Feedback Loop

**User Story:** As a user, I want to rate AI advice as helpful or unhelpful, so that the system learns my preferences and improves over time.

#### Acceptance Criteria

1. THE System SHALL allow a user to submit a thumbs-up or thumbs-down rating for any AI advice item via `PATCH /api/advice/:id/feedback`.
2. WHEN a user marks an advice item as implemented, THE System SHALL record the implemented flag and timestamp in the `ai_advice` table.
3. THE AIGrowthAdvisor SHALL reduce the frequency of advice of the same type for a user if that advice type has received thumbs-down ratings on 3 or more consecutive instances.
4. THE System SHALL allow a user to request an explanation of any advice item, and THE AIGrowthAdvisor SHALL respond with a natural language explanation referencing the underlying metrics.

### Requirement 21: Scenario Modeling

**User Story:** As a founder, I want to model "what if" scenarios, so that I can project the impact of strategic changes before committing resources.

#### Acceptance Criteria

1. THE ScenarioModeler SHALL accept a user-defined input specifying a target metric and a hypothetical new value (e.g., "D7 retention = 25%").
2. WHEN a scenario is submitted, THE ScenarioModeler SHALL project the estimated impact on active wallet count, monthly revenue, and 90-day growth rate based on patterns from similar contracts in the same category.
3. THE ScenarioModeler SHALL return scenario results within 15 seconds of the request.
4. THE System SHALL expose scenario modeling via `POST /api/scenarios`.

### Requirement 23: JavaScript SDK

**User Story:** As a developer, I want a JavaScript SDK, so that I can integrate MetaGauge analytics into my application with minimal boilerplate.

#### Acceptance Criteria

1. THE SDK SHALL be published as an npm package named `@metagauge/sdk`.

### Requirement 24: Category Benchmarking

**User Story:** As a founder, I want to see how my protocol's metrics compare to the category average, so that I know whether my performance is strong or weak relative to peers.

#### Acceptance Criteria

1. THE BenchmarkingService SHALL group all indexed contracts by category (DeFi, NFT, DAO, Gaming, Infrastructure) and calculate the following statistics for each metric: average, 25th percentile, 50th percentile, 75th percentile, and 90th percentile.
2. THE BenchmarkingService SHALL recalculate category benchmarks on a schedule no less frequent than once every 24 hours.
3. THE BenchmarkingService SHALL store benchmark results in the `category_benchmarks` table with fields: category, metric_name, avg_value, p25_value, p50_value, p75_value, p90_value, sample_size, calculated_at.
4. THE System SHALL expose category benchmarks via `GET /api/benchmarks/:category`.
5. THE Dashboard SHALL display the user's contract metric value alongside the category average and the user's percentile rank for each key metric.
6. IF a category has fewer than 3 contracts, THEN THE BenchmarkingService SHALL not calculate or expose benchmarks for that category to avoid misleading comparisons.

### Requirement 25: Data Accuracy Monitoring

**User Story:** As a platform operator, I want the system to monitor its own indexing health, so that data quality issues are detected and surfaced automatically.

#### Acceptance Criteria

1. THE System SHALL monitor the time elapsed since the last successful indexing cycle for each registered contract.
2. WHEN a contract has not been successfully indexed for more than 2 hours, THE System SHALL generate an internal data accuracy alert and log it.
3. THE System SHALL expose indexing health status for all contracts via `GET /api/admin/indexing-health` (Admin role only).
4. THE System SHALL record all configuration changes and user actions in an audit log stored in the database.
5. THE System SHALL expose the audit log via `GET /api/admin/audit-log` (Admin role only), returning entries sorted by timestamp descending.

---

### Requirement 26: Parser and Serializer Round-Trip Integrity

**User Story:** As a developer, I want all data serialization and deserialization to be lossless, so that exported data exactly matches what is stored and displayed.

#### Acceptance Criteria

1. THE System SHALL parse ABI JSON strings into structured ABI objects when a contract is registered.
2. WHEN an ABI JSON string is parsed and then serialized back to JSON, THE Serializer SHALL produce a JSON string that parses to an equivalent ABI object (round-trip property).
3. THE System SHALL parse blockchain transaction data from RPC responses into internal transaction objects.
4. WHEN a transaction object is serialized to CSV and the CSV row is parsed back into a transaction object, THE resulting object SHALL have the same field values as the original (round-trip property for CSV export).
5. IF an ABI JSON string is malformed, THEN THE Parser SHALL return a descriptive error message identifying the location of the parse failure.
